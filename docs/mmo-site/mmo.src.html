맵당 동접 4,000, 병목 개선 R&D

패킷이 흐르는 공장 — 서버 구조를 위에서 내려다봅니다. 왼쪽 아래(접속)에서 오른쪽 위(저장)로 흐르고, 건물을 누르면 그 부분의 설계·구현 상세가 열립니다.

수신 recv 응답 send 저장 DB 환승 = 스레드 경계 (큐 인계) ▸ 건물 클릭 · 화물 흐르는 중
시간 흐름 = 왼쪽 아래 → 오른쪽 위
읽는 법 왼쪽 아래(접속) → 중앙(게임 루프 단일 코어) → 오른쪽 위(저장). 응답은 앞 레인으로 접속 군중에게 되돌아갑니다. 한 줄 단일 스레드가 섬처럼 홀로 서고, 네트워크·저장 워커가 그 코어의 시간을 지키기 위해 주변을 돕니다.

가설을 세우고 실측으로 채택/기각한 병목 개선 실험입니다. 수치는 전부 Prometheus/Grafana 실측값 — 실측 없이는 병목이라 단정하지 않았습니다.

동접 4,000을 만드는 부하 테스트 더미 클라이언트와 측정 방법. "주장 대신 실측"의 실측이 여기서 나옵니다.

open-loop 부하 주입
응답을 기다리지 않고 요청을 계속 밀어넣습니다. RTT가 올라가도 부하는 유지되므로 RTT는 은폐지표가 아니라 서버 송신 포화의 신호로 읽습니다.
코어 격리 (affinity)
서버는 물리코어 0–5, 더미 클라는 물리코어 6–9에 고정해 한 PC 안에서 crosstalk를 차단합니다. 반드시 물리코어 단위로 마스크를 잡습니다(논리코어 오독 금지).
건강 판정 4종
loop_p99 < 40ms · buffer_full = 0 · session = 목표치 · flush_backlog 선형 — 이 넷이 유지되는 선까지가 건강한 동접(물리코어당 ~1,375).
무인 스윕 자동화
WT×K 교차 스윕 · ClientCount 천장 스윕 · A/B 무인 툴(OFF/ON 리빌드 → 수집 → 비교)로 사람 개입 없이 실측 CSV를 뽑습니다.

스택과 링크.

스택 · 계측
C++17 · Windows IOCP · MySQL(비동기 저장) · Prometheus + Grafana 상시 계측(tick · CPU · 송신 지표).
천장
